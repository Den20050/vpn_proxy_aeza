# ТЗ для Claude Sonnet 4.6 — усиленная версия

## Проект
Развёртывание устойчивого к блокировкам РКН (ТСПУ) VPN/прокси-сервера на базе VLESS + Reality + Sing-box.

## Цель задачи
Создать полную, пошаговую, production-ready инструкцию и все необходимые файлы для развёртывания Sing-box (VLESS + Reality) на VPS в Нидерландах.

Параметры сервера:
- Тариф: NLs-1
- 1 ядро CPU
- 2 ГБ RAM
- 30 ГБ NVMe
- Ubuntu 22.04

Сервер должен одновременно обслуживать:
- Telegram-бота Silent Couple Bot (aiogram 3 + arq + Redis queue)
- 10-15 обычных пользователей (с возможностью масштабирования до 20-25)

---

## Требования к решению

### Устойчивость к блокировкам
- Использовать VLESS + Reality с uTLS fingerprint Chrome
- Реализовать маскировку под популярные сайты (рекомендовать несколько вариантов: `microsoft.com`, `apple.com`, `cloudflare.com` и т.д.)
- Использовать порт `443` (TCP + UDP при необходимости)
- Поддержать `xtls-rprx-vision` flow

### Архитектура использования
- Основной inbound: VLESS + Reality для пользователей (отдельный UUID на человека или на небольшую группу)
- Дополнительный inbound: SOCKS5 (локальный, `127.0.0.1:1080`) специально для Telegram-бота (selective routing)
- Бот размещён на этом же VPS (предпочтительный вариант); Timeweb-вариант описать отдельным разделом как fallback

### Производительность и ограничения
- Сервер имеет всего 1 ядро и 2 ГБ RAM — решения должны быть максимально лёгкими
- Включить mux (multiplexing) для снижения нагрузки
- Добавить разумные лимиты на соединения
- Учесть пиковую нагрузку утром (07:30) и вечером (21:30), когда бот массово отправляет картинки

## Критерии приёмки (Acceptance Criteria)

Результат считается готовым, если:

| # | Критерий | Проверка |
|---|----------|----------|
| 1 | Sing-box запускается без ошибок | `systemctl status singbox` → `active (running)` |
| 2 | VLESS + Reality принимает подключения на порту 443 | `sing-box check` + тест клиентом (v2rayN / Hiddify) |
| 3 | SOCKS5 локальный proxy поднят | `curl --socks5 127.0.0.1:1080 https://api.telegram.org` возвращает 200 |
| 4 | Telegram-бот доступен через SOCKS5 (selective routing) | Бот отправляет картинку без ошибок в пиковое время |
| 5 | RAM под Sing-box не превышает 150 МБ в покое | `ps aux` / `htop` |
| 6 | CPU под нагрузкой 10 пользователей не превышает 60% | `htop` при одновременном стриминге |
| 7 | Firewall разрешает только 22, 443; всё остальное DROP | `ufw status verbose` |
| 8 | Конфиг валиден через встроенную проверку | `sing-box check -c /etc/sing-box/config.json` → 0 ошибок |
| 9 | Скрипт генерации ключей выдаёт валидные vless:// ссылки и QR | Проверяется импортом в клиент |
| 10 | Все конфиги и UUID сохранены в `/root/vpn-backup/` | `ls -la /root/vpn-backup/` |

## SLA / SLO (целевые показатели)

| Метрика | Цель | Критический порог |
|---------|------|-------------------|
| Доступность сервиса (uptime) | ≥ 99% в месяц | < 95% → расследование |
| Задержка (RTT клиент → сервер) | ≤ 80 мс | > 200 мс → смена `server_name` |
| RAM usage (Sing-box + бот + Redis) | ≤ 1.5 ГБ | > 1.8 ГБ → оптимизация |
| CPU idle в покое | ≥ 60% | < 30% → расследование нагрузки |
| Время старта после перезагрузки | ≤ 30 сек | > 60 сек → проверка `systemd` |

## Политика ротации ключей и UUID

Документ должен включать регламент со следующими правилами:

### Плановая ротация
- `short_id` — менять раз в 3 месяца или при любом признаке деградации соединения
- `server_name` — менять при появлении жалоб на сброс соединений; иметь список из 3-5 резервных вариантов
- UUID пользователей — менять при отзыве доступа или компрометации; каждый пользователь должен иметь свой UUID

### Экстренная ротация (при блокировке)
- Шаг 1: Сменить `server_name` → перезапустить Sing-box (без простоя клиентов после реконнекта)
- Шаг 2: Сменить `short_id` (требует обновления у всех клиентов)
- Шаг 3: При полной блокировке IP — сменить VPS / добавить резервный сервер

Скрипт ротации должен:
1. Генерировать новые значения
2. Обновлять `config.json`
3. Перезапускать `sing-box`
4. Выводить новые ссылки и QR-коды

## Резервное копирование (Backup)

Документ должен описывать стратегию бэкапов:

### Что бэкапить
- `/etc/sing-box/config.json`
- `/root/vpn-backup/` (ключи, UUID, ссылки пользователей)
- `.env` / конфигурационные файлы Telegram-бота
- Дамп Redis (если состояние очереди критично)

### Как бэкапить
- Локально: ежедневный `cron`-скрипт, архивирующий указанные файлы в `/root/backups/YYYY-MM-DD/`
- Удалённо: опциональная отправка через `scp` или `rclone` на внешний storage (Timeweb / S3-совместимый)
- Ретенция: хранить последние 7 ежедневных и 4 еженедельных бэкапа

### Восстановление
Скрипт или инструкция должны описывать полное восстановление с нуля (clean VPS → работающий Sing-box + бот) за ≤ 20 минут.

## Runbook: действия при блокировках

Документ должен содержать раздел с пошаговыми действиями для следующих сценариев:

### Сценарий 1: Замедление / нестабильные соединения
1. Проверить `journalctl -u singbox -n 100`
2. Проверить RTT командой `ping` / `mtr` до `server_name`
3. Сменить `server_name` на резервный из списка → перезапуск
4. Если не помогло — сменить `short_id` + переслать новые ссылки пользователям

### Сценарий 2: Полная недоступность (IP заблокирован)
1. Подтвердить блокировку: `curl -m 5 https://77.73.135.202:443` с другого IP
2. Уведомить пользователей через резервный канал
3. Поднять новый VPS (заготовить инструкцию `one-command-deploy.sh`)
4. Мигрировать из бэкапа

### Сценарий 3: Перегрузка сервера (OOM / CPU 100%)
1. `journalctl -u singbox` + `htop` → найти источник
2. Убить лишние соединения: `sing-box` reload
3. Уменьшить `max_connections` в конфиге
4. При регулярности — планировать апгрейд тарифа

### Сценарий 4: Telegram-бот не отправляет сообщения
1. Проверить доступность SOCKS5: `curl --socks5 127.0.0.1:1080 https://api.telegram.org`
2. Проверить Redis: `redis-cli ping`
3. Проверить `arq` worker: `systemctl status bot-worker`
4. Проверить routing — убедиться, что трафик бота идёт через SOCKS5

## Что нужно подготовить

### Основные файлы
- Полный `config.json` для Sing-box (с комментариями к каждому параметру)
- `singbox.service` — systemd unit-файл
- `generate-keys.sh` — скрипт генерации ключей, UUID, vless:// ссылок и QR-кодов
- `rotate.sh` — скрипт экстренной/плановой ротации `short_id` и `server_name`
- `backup.sh` — cron-скрипт резервного копирования
- Инструкция по firewall (`ufw`)
- Selective routing для Telegram-бота (через SOCKS5 или policy routing)
- Рекомендации по мониторингу (`htop`, `iftop`, `vnstat`, `journalctl`)

### Дополнительно (по возможности)
- `update-singbox.sh` — скрипт автоматического обновления Sing-box
- `one-command-deploy.sh` — скрипт полного развёртывания с нуля
- Таблица резервных `server_name` с оценкой надёжности
- План апгрейда сервера при росте до 25+ пользователей
